# `hoauth2` module dependencies

Topological order of every module of the [`hoauth2`](https://hackage.haskell.org/package/hoauth2)
package (v2.15.1, source: https://github.com/freizl/hoauth2) imported into
`linen`, per [AGENTS.md](../../../AGENTS.md)'s Hackage-import convention.

An edge **A → B** means *module A imports module B*, so **B must be built
before A**.

## External (non-`hoauth2`) dependencies

Resolved against `hoauth2-2.15.1.cabal`'s `build-depends`, in Hackage-import
precedence order (Lean stdlib > existing `linen` Haskell port > new source).

Already ported, reused as-is (no new work):

- `aeson` → `Aeson`
- `base`, `bytestring`, `containers`, `text`, `transformers` → `Base`,
  `ByteString`, `Containers`, `Text`, Lean's own `ExceptT`/`ReaderT`/`StateT`
- `base64` → `Base64`
- `data-default` → `DataDefault`
- `http-conduit` → `HttpConduit`
- `http-types` → `HttpTypes`

Substituted rather than freshly ported (each a genuine precedence-rule
application, not a shortcut around real work — see the per-package note):

- **`memory`** (`Data.ByteArray`) — used only as a generic byte-buffer
  abstraction; Lean's native `ByteArray` already *is* the concrete
  representation every instance in this port would reduce to, so there is
  nothing left to port.
- **`exceptions`** (`Control.Monad.Catch`) — `Network.OAuth2.Internal`'s one
  call site (`uriToRequest :: MonadThrow m => URI -> m Request`) is a bare
  `MonadThrow` constraint with no actual `catch`/`handle` logic anywhere in
  `hoauth2`; ported as a plain `IO`/`Except`-returning function using
  `Linen.Control.Exception` (already in the library), per this package's own
  Hackage-import precedence note.
- **`microlens`** (`Lens.Micro`) — exactly three call sites across the whole
  package (`Network.OAuth2.Internal`, `.HttpClient`,
  `.AuthorizationRequest`), all of the shape
  `over (queryL . queryPairsL) f uri` (append extra query parameters to a
  URI). Porting a general lens abstraction for three call sites would be the
  "abstraction the task doesn't need" AGENTS.md/CLAUDE.md's coding
  conventions warn against; ported as one direct helper function on this
  port's own URI type instead.
- **`uri-bytestring` + `uri-bytestring-aeson`** — `linen` already has a
  ported URI type, `Linen.Network.URI` (from the already-imported
  `network-uri`, see the top-level import index). Per the Hackage-import
  precedence rule ("check whether \[it\] already exists ... in this order:
  Lean stdlib > Haskell/Hackage \[already in `linen`\] > new source"),
  `hoauth2`'s `URI`/`URIRef` usage is re-pointed at `Linen.Network.URI`
  rather than importing a second, overlapping URI library. The
  `uri-bytestring-aeson` JSON instance and the `URI.ByteString.QQ`
  quasiquoter (compile-time literal syntax, no Lean macro-system analogue
  needed here — call sites just become ordinary parsed string literals) are
  written directly inside the `hoauth2` port itself rather than as a
  separate package.
- **`binary` + `binary-instances`** — used for exactly one thing: an
  `instance Binary TokenResponse` whose `put`/`get` just proxies through the
  *already-derived* `ToJSON`/`FromJSON` value (`put rawResponse` /
  `fromJSON (Aeson.Object rawt)`). This is a GHC-`Data.Binary`-specific
  disk-serialization convenience layered on top of JSON that adds no OAuth2
  semantics of its own — out of scope the same way `deepseq` is excluded from
  the `hip` entry (a GHC-runtime-specific facility with no bearing on the
  behavior being ported); callers needing to persist a `TokenResponse` use
  the already-ported JSON `toJSON`/`fromJSON` directly.
- **`crypton`** (`Crypto.Hash`, `Crypto.Random`) — used in exactly one module,
  `Experiment.Pkce`, for the two primitives PKCE's `S256` challenge method
  needs: a plain SHA-256 digest (`code_challenge = base64url(sha256(code_verifier))`)
  and a CSPRNG byte generator (`code_verifier` itself). Porting all of
  `crypton` (dozens of hash/cipher/KDF algorithms, none else used here) for
  two primitives would repeat the exact over-import `duckdb-ffi`'s scope note
  and `hip`'s `Chart`/`Histogram` exclusion both already reject. `linen`
  already links OpenSSL for `Crypto.JOSE.FFI`/`ffi/jose.c`
  (HMAC/RSA/EC-verify), which exposes both `EVP_sha256`-family digests and
  `RAND_bytes` — this port adds two small new `@[extern]` bindings reusing
  that same already-linked OpenSSL, in a new `Linen.Crypto.SHA256`
  (digest) and `Linen.Crypto.SecureRandom` (CSPRNG bytes) module, instead of
  a full `crypton` import. No `lakefile.lean` changes needed (OpenSSL
  discovery already exists for `jose.o`).

replaced with `linen`'s own version string.

## Native C library

No new native dependency: the two new FFI primitives (SHA-256 digest,
`RAND_bytes` CSPRNG) reuse OpenSSL, already linked for
`Crypto.JOSE.FFI`/`ffi/jose.c` — implemented as two new functions inside that
same `ffi/jose.c` shim, no `lakefile.lean` change needed.
